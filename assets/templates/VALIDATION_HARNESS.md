# Validation Harness: {{APP_NAME}}

## 공통 검증

```bash
npm run lint
npm run build
```

왜 하는가:
- lint는 코드 품질과 명백한 실수를 잡는다.
- build는 실제 배포 전에 타입/라우팅/번들 오류를 잡는다.

## 인증 검증

| 상황 | 기대 결과 |
| --- | --- |
| 미로그인으로 공개 랜딩 접근 | 200 |
| 미로그인으로 보호 상세 접근 | `/login?next=...` |
| 로그인 성공 | 원래 경로 복귀 |
| 로그아웃 후 보호 상세 접근 | 다시 `/login?next=...` |

## 데이터 검증

- [ ] 필요한 테이블이 존재한다.
- [ ] RLS가 켜져 있다.
- [ ] 본인 데이터만 읽고 쓸 수 있다.
- [ ] 공개 데이터와 개인 데이터가 분리되어 있다.
- [ ] service role 키가 클라이언트에 노출되지 않는다.

## 배포 검증

- [ ] Vercel Production env가 설정되어 있다.
- [ ] 운영 URL이 200을 반환한다.
- [ ] Supabase Redirect URL이 운영 콜백과 일치한다.
- [ ] 운영에서 로그인 후 원래 경로로 돌아온다.
- [ ] 500 오류가 있으면 배포 로그를 먼저 확인한다.

## 페이즈 통과 규칙

- 검증 실패가 있으면 다음 페이즈로 넘어가지 않는다.
- 실패 원인은 문서에 기록한다.
- 임시 우회는 `KNOWN_ISSUES.md`나 해당 페이즈 노트에 남긴다.
